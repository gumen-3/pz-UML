# Practical lesson pz-UML
Побудова повердінкових UML-діаграм для проєктування інформаційних систем

## Тема - Програмний модуль підтримки прийняття рішень при розподілі випускників ВВНЗ 

## 1. Діаграма варіантів використання (Use Case Diagram)

Ця діаграма описує функціональність системи з точки зору користувачів (акторів).

* **Випускник:** має право переглядати список посад та вказувати свої побажання.
* **Офіцер кадрів:** завантажує дані, запускає алгоритм розрахунку рейтингу та формує попередні списки.
* **Комісія:** приймає остаточне рішення на основі згенерованих даних.

`code`

    graph LR
    Graduate((Випускник))
    HR_Officer((Офіцер кадрів))
    Admin((Адміністратор))
    Commission((Комісія з розподілу))

    subgraph "Модуль розподілу випускників"
        UC1(Перегляд доступних посад)
        UC2(Подання пріоритетів вибору)
        UC3(Введення даних про вакансії)
        UC4(Розрахунок рейтингу)
        UC5(Генерація варіантів розподілу)
        UC6(Затвердження фінального списку)
        UC7(Керування обліковими записами)
    end

    Graduate --> UC1
    Graduate --> UC2
    
    HR_Officer --> UC3
    HR_Officer --> UC4
    HR_Officer --> UC5
    
    Commission --> UC6
    
    Admin --> UC7

    UC5 -.-> |include| UC4
    UC6 -.-> |include| UC5

`code`

## 2. Діаграма послідовності (Sequence Diagram)

Відображає процес взаємодії об'єктів у часі для сценарію "Автоматизований розподіл за рейтингом".

Діаграма демонструє шлях від запиту кадровика до отримання готового проекту наказу. Основна логіка виконується в "Модулі ППР", який зіставляє кращих випускників з найбільш пріоритетними посадами.

`code`

    sequenceDiagram
    participant HR as Офіцер кадрів
    participant System as Інтерфейс Системи
    participant Engine as Модуль ППР (Логіка)
    participant DB as База Даних

    HR->>System: Запит на формування розподілу
    System->>DB: Отримати список випускників та вакансій
    DB-->>System: Дані (рейтинги, посади)
    
    System->>Engine: Розрахувати оптимальний розподіл
    activate Engine
    Note over Engine: Алгоритм враховує рейтинг <br/>та пріоритети
    Engine-->>System: Список рекомендованих призначень
    deactivate Engine
    
    System-->>HR: Відобразити проект розподілу
    
    HR->>System: Підтвердити та зберегти
    System->>DB: Оновити статус посад/випускників
    DB-->>System: Успішно збережено
    System-->>HR: Сформовано наказ про розподіл
    
`code`

## 3. Діаграма діяльності (Activity Diagram)

Описує алгоритм роботи системи від початку збору даних до фінального затвердження.

Діаграма показує логіку прийняття рішень: якщо автоматичний розподіл не задовольняє Комісію (наприклад, через специфічні сімейні обставини, не враховані системою), передбачено цикл ручного коригування перед фінальним друком документів.

`code`

    flowchart TD
    Start([Початок процесу розподілу]) --> InputData[Збір даних: рейтинг випускників та перелік вакансій]
    InputData --> CheckData{Дані повні?}
    
    CheckData -- Ні --> RequestMore[Запит відсутньої інформації]
    RequestMore --> InputData
    
    CheckData -- Так --> RunAlgo[Запуск алгоритму ППР]
    
    RunAlgo --> ShowResults[Представлення результатів Комісії]
    
    ShowResults --> CommissionDecision{Затвердити?}
    
    CommissionDecision -- Ні (коригування) --> ManualEdit[Ручне внесення змін кадровиком]
    ManualEdit --> ShowResults
    
    CommissionDecision -- Так --> GenerateDocs[Генерація витягів із наказу та приписів]
    GenerateDocs --> End([Кінець процесу])
    
`code`

### Побудовані діаграми охоплюють ключові аспекти проектованої системи:

* **Use Case:** Хто і що робить.

* **Sequence:** Як компоненти системи взаємодіють між собою для досягнення результату.

* **Activity:** Послідовність кроків та розгалуження в бізнес-процесі розподілу.

Всі діаграми логічно узгоджені: актори з Use Case згадуються в Sequence, а етапи діяльності відповідають функціям системи.
